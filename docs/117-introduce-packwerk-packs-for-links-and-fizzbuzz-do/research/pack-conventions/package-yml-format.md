# package.yml Format Reference

## Fields

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `enforce_dependencies` | bool / `"strict"` | `false` | Require explicit dependency declarations |
| `enforce_privacy` | bool / `"strict"` | `false` | Limit external access to `app/public/` (requires packwerk-extensions) |
| `enforce_layers` | bool / `"strict"` | `false` | Enforce architectural layer ordering (requires packwerk-extensions) |
| `layer` | string | (none) | Which architecture layer this pack belongs to |
| `public_path` | string | `"app/public/"` | Override default public API directory |
| `dependencies` | list of pack paths | `[]` | Packs this pack may reference |
| `metadata` | map | `{}` | Free-form ownership or contact info |

## Root package.yml

Located at `./package.yml` (sibling of `Gemfile`). Represents the Rails app
itself — the "default" pack that contains everything not yet moved into a
named pack.

```yaml
# package.yml (root)
enforce_dependencies: false
enforce_privacy: false
```

Keep enforcement off at root. The surveys domain, infrastructure classes, and
workbook stubs remain here and were never subject to boundary rules. Turning on
enforcement here would require auditing all that code first.

## packs/fizzbuzz/package.yml

```yaml
# packs/fizzbuzz/package.yml
enforce_dependencies: true
enforce_privacy: true

dependencies:
  - "."                 # root package (ApplicationController, ApplicationJob, etc.)
  # - "packs/links"    # uncomment if fizzbuzz needs to reference links public API
```

`"."` refers to the root package. All packs that use Rails infrastructure
(ApplicationController, ApplicationRecord, ApplicationJob) need this
dependency.

## packs/links/package.yml

```yaml
# packs/links/package.yml
enforce_dependencies: true
enforce_privacy: true

dependencies:
  - "."                 # root package (ApplicationController, ApplicationJob, etc.)
```

## enforce_dependencies values

| Value | Behavior |
|-------|----------|
| `false` | No dependency checking |
| `true` | Undeclared cross-package references are violations |
| `"strict"` | Like `true`, but also treats any entry in `package_todo.yml` as a violation |

Start with `true` for new packs. Use `"strict"` once the pack is stable and
confirmed violation-free.

## enforce_privacy values (requires packwerk-extensions)

| Value | Behavior |
|-------|----------|
| `false` | No privacy checking |
| `true` | References to private constants (outside `app/public/`) from other packs are violations |
| `"strict"` | Like `true`, but also treats `package_todo.yml` entries as violations |

## Public API Declaration

With `enforce_privacy: true`, a pack exposes only constants under
`app/public/`. Any constant in `app/models/`, `app/controllers/`, etc. is
private — other packs cannot reference it without declaring a dependency AND
the constant must be in `app/public/`.

For fizzbuzz_app, neither `packs/links` nor `packs/fizzbuzz` needs to expose
constants to each other (the QrCodeGenerator cross-dependency is resolved by
moving it to root). Both packs can set `enforce_privacy: true` with empty
`app/public/` directories.

## package_todo.yml Strategy

`package_todo.yml` records known/pre-existing violations so `bin/packwerk check`
passes despite them. Generated by `bin/packwerk update-todo`.

**Do not use for new packs.** Since `packs/links` and `packs/fizzbuzz` are
created from scratch, start with zero violations. If `bin/packwerk check`
reports violations, fix the code — do not record them in a todo file.

## Architecture Layers (packwerk-extensions)

The `enforce_layers` checker enforces a declared hierarchy of pack layers.
Layers are defined in `packwerk.yml`; each `package.yml` declares which layer
it belongs to.

### packwerk.yml — declare the layer hierarchy

```yaml
# packwerk.yml
architecture_layers:
  - feature      # domain packs (highest — may depend on lower layers)
  - utility      # shared infrastructure (lowest — may not depend on feature packs)
```

Layers are listed from highest to lowest. A pack in a higher layer MAY depend
on packs in the same or lower layers. A pack in a lower layer MUST NOT depend
on packs in a higher layer.

### Layer assignments for fizzbuzz_app

```
Layer: feature    → packs/fizzbuzz, packs/links
Layer: utility    → root package (ApplicationRecord, ApplicationController, etc.)
```

This means:
- Feature packs (fizzbuzz, links) CAN depend on the root utility layer ✓
- The root utility layer CANNOT depend on feature packs ✓
- Two feature packs CANNOT depend on each other without `enforce_layers` blocking
  it (but `enforce_dependencies: true` already prevents this via missing declaration)

### package.yml — declare layer membership

```yaml
# package.yml (root)
enforce_dependencies: false
enforce_privacy: false
enforce_layers: true
layer: utility
```

```yaml
# packs/fizzbuzz/package.yml
enforce_dependencies: true
enforce_privacy: true
enforce_layers: true
layer: feature
dependencies:
  - "."
```

```yaml
# packs/links/package.yml
enforce_dependencies: true
enforce_privacy: true
enforce_layers: true
layer: feature
dependencies:
  - "."
```

### enforce_layers values

| Value | Behavior |
|-------|----------|
| `false` | No layer checking |
| `true` | References that violate layer order are violations |
| `"strict"` | Like `true`, but also treats `package_todo.yml` entries as violations |

## Metadata Field (optional)

Use for ownership information:

```yaml
# packs/links/package.yml
enforce_dependencies: true
enforce_privacy: true
dependencies:
  - "."
metadata:
  owner: "platform-team"
```

Packwerk ignores unknown keys; this is purely for human documentation.

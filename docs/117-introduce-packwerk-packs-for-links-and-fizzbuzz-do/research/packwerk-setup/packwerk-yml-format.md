# Packwerk Configuration Format

## packwerk.yml Reference

`packwerk.yml` lives at the Rails root. Generate it with `bin/packwerk init` (see [bin/packwerk Generation](#binpackwerk-generation)).

### All Fields

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `include` | glob list | `**/*.{rb,rake,erb}` | Files to analyze |
| `exclude` | glob list | `{bin,node_modules,script,tmp,vendor}/**/*` | Files/dirs to skip |
| `associations_exclude` | glob list | (none) | Paths excluded from association inspection |
| `package_paths` | glob or list | `**/` | Where to look for `package.yml` files |
| `custom_associations` | list | (none) | Extra ActiveRecord association names to track |
| `parallel` | boolean | `true` | Fork parsing into subprocesses |
| `cache` | boolean | `false` | Cache parsed file results |
| `cache_directory` | path | `tmp/cache/packwerk` | Cache storage location |
| `require` | list | (none) | Ruby files/gems to load before analysis (used for extensions) |
| `offenses_formatter` | string | (default) | Custom formatter class identifier |

### Removed Fields (do not add)

- **`load_paths`** — removed in packwerk 2.0. Packwerk now reads autoload paths directly from the Rails app via Zeitwerk. Do not add this field; it will be silently ignored.
- **`inflections_file`** — removed in packwerk 3.0. Use the standard Rails `config/initializers/inflections.rb` instead.

### Recommended packwerk.yml for fizzbuzz_app

```yaml
# packwerk.yml
# Generated by bin/packwerk init, then customized.

# Patterns of files/dirs to include in analysis.
# include:
# - "**/*.{rb,rake,erb}"

# Patterns of files/dirs to exclude from analysis.
# exclude:
# - "{bin,node_modules,script,tmp,vendor}/**/*"

# Glob pattern(s) to find package.yml files.
# Default "*/**/" finds packages anywhere in the tree.
# Restrict to just packs/ to keep things explicit:
package_paths:
  - "./"
  - "packs/*/"

# Load extension gems before analysis.
# Required if using packwerk-extensions for privacy/visibility enforcement.
require:
  - packwerk-extensions

# When true, Packwerk forks subprocesses for parsing (faster on multi-core).
parallel: true

# Cache parsed results for incremental check runs (useful in CI).
cache: true
cache_directory: tmp/cache/packwerk
```

**Notes on `package_paths`:**
- `"./"` includes the root package (the Rails app itself)
- `"packs/*/"` finds `packs/links/`, `packs/fizzbuzz/`, etc.
- Default `"**/"` would also find any nested `package.yml` in `app/` subdirectories — being explicit avoids surprise

## Root package.yml

The root `package.yml` (at `./package.yml`, sibling of `Gemfile`) represents the Rails app itself. `bin/packwerk init` creates it.

### Minimum content (enforcement off at root)

```yaml
# package.yml (root)
# The root package represents the Rails application itself.
# Keep enforcement off here — new code lives in packs, not the root.
enforce_dependencies: false
enforce_privacy: false
```

**Rationale:** The root package contains existing app code that predates Packwerk. Setting enforcement to `false` means violations from legacy root code are not reported, letting you introduce packs incrementally without an initial flood of errors. New packs (`packs/links/`, `packs/fizzbuzz/`) start with enforcement on.

### Root package.yml with enforcement on (stricter, for greenfield)

If the app root is already clean and you want full enforcement immediately:

```yaml
enforce_dependencies: true
enforce_privacy: false  # Privacy lives in individual packs; root is the implicit "default" pack
```

For fizzbuzz_app (small app, new packs from scratch), starting with `enforce_dependencies: false` at root is safest.

## Individual Pack package.yml

Each pack under `packs/` needs its own `package.yml`:

```yaml
# packs/fizzbuzz/package.yml
enforce_dependencies: true
enforce_privacy: true  # requires packwerk-extensions
dependencies:
  - "."  # can reference root package constants
```

```yaml
# packs/links/package.yml
enforce_dependencies: true
enforce_privacy: true
dependencies:
  - "."
  # - "packs/fizzbuzz"  # add if links depends on fizzbuzz
```

### enforce_dependencies values

| Value | Meaning |
|-------|---------|
| `false` | No dependency checking |
| `true` | Undeclared cross-package references are violations |
| `"strict"` | Like `true`, but also forbids exceptions recorded in `package_todo.yml` |

### enforce_privacy values (requires packwerk-extensions)

| Value | Meaning |
|-------|---------|
| `false` | No privacy checking |
| `true` | References to private constants (outside `app/public/`) are violations |
| `"strict"` | Like `true`, but also forbids exceptions recorded in `package_todo.yml` |

### Public API convention

With `enforce_privacy: true`, each pack exposes its public API via `app/public/`:

```
packs/fizzbuzz/
  app/
    public/         # public API (accessible by other packs)
      fizzbuzz.rb
    models/         # private (FizzBuzz internals)
      ...
  package.yml
```

## bin/packwerk Generation

The `bin/packwerk` binstub is NOT auto-generated by the gem install. Generate it explicitly:

```sh
bundle binstub packwerk
```

This creates `bin/packwerk` (a thin wrapper that invokes the gem's CLI). Commit this file.

Then initialize Packwerk's configuration files:

```sh
bin/packwerk init
```

This creates:
- `packwerk.yml` — configuration (all options commented out; uncomment as needed)
- `package.yml` — root package (with `enforce_dependencies: false` and `enforce_privacy: false`)

## validate vs. check

These are distinct commands with different purposes:

### `bin/packwerk validate`

**Purpose:** Verifies that the Packwerk configuration and package graph are valid. Does not check for violations.

**What it checks:**
- All `package.yml` files are syntactically valid
- All declared `dependencies:` reference packages that actually exist
- The dependency graph is acyclic (no circular dependencies)
- Autoload paths are correctly extracted from the Rails app (boots Rails)

**When to run:** On CI, before `check`; also run locally after adding a new pack or changing dependencies.

**Failure means:** Configuration error — something is wrong with the pack setup itself, not with application code.

```sh
bin/packwerk validate
```

### `bin/packwerk check`

**Purpose:** Scans application code for boundary violations against the declared package rules.

**What it checks:**
- References to constants in other packages that are not declared as dependencies
- References to private constants in other packages (if `enforce_privacy` is on and packwerk-extensions is loaded)

**When to run:** On every CI run (after `validate`); optionally in a git pre-commit hook.

**Failure means:** Application code violates the declared package boundaries.

```sh
bin/packwerk check              # check all files
bin/packwerk check packs/fizzbuzz  # check only the fizzbuzz pack
```

### Recommended CI order

```sh
bin/packwerk validate && bin/packwerk check
```

## package_todo.yml Strategy

`package_todo.yml` files record known/pre-existing violations so that `bin/packwerk check` passes despite them. They are generated by:

```sh
bin/packwerk update-todo
```

### Strategy for fizzbuzz_app (new packs, zero violations)

**Do not use `package_todo.yml` for the new packs.** Since `packs/links` and `packs/fizzbuzz` are being created from scratch as part of Issue #117, there are no pre-existing violations. Start with enforcement on and zero todo files.

The correct workflow:
1. Create `packs/fizzbuzz/` and `packs/links/` with `enforce_dependencies: true`
2. Move code into the packs
3. Run `bin/packwerk check` — fix any violations before committing
4. Never run `bin/packwerk update-todo` to paper over a violation

**When `package_todo.yml` IS appropriate:**
- Migrating a large existing codebase where fixing all violations immediately is impractical
- The USAGE.md guidance: "The package TODO list should not be added to, but worked off over time"

### If you must allow a temporary violation

Add it manually to the offending pack's `package_todo.yml` with a comment and a tracking issue, and immediately open a follow-up issue to remove it.

## Zeitwerk Integration

Packwerk does not configure Zeitwerk — it reads from it. The requirement is that `packs/*/app/` directories are registered as autoload paths before `bin/packwerk validate` runs (which boots Rails).

Add to `config/application.rb` (if not using `packs-rails`):

```ruby
# config/application.rb
module FizzBuzzApp
  class Application < Rails::Application
    # ... existing config ...

    # Register packs as autoload roots so Zeitwerk discovers their constants.
    config.paths.add "packs", glob: "*/app/{*,*/concerns}", eager_load: true
  end
end
```

Verify Zeitwerk is happy after adding this:

```sh
bin/rails zeitwerk:check
```
